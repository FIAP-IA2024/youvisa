# Sprint 4: Multi-Agent + Customer Portal — Plan

**For this spec:** `[[spec]]`

**Goal:** Replace YOUVISA's monolithic NLP Lambda with a typed multi-agent pipeline running locally via Claude Agent SDK, remove n8n entirely, migrate to local Mongo + MinIO, build a JWT-authenticated customer portal, and refresh the operator console UI/UX — all without regression of Sprint 2/3 behavior.

## Approach

The work follows three principles:

**1. Local-first stack with one `docker compose up`.** The new compose file orchestrates six services: `mongo:7`, `minio/minio`, `agent` (Node + Claude SDK), `api` (Node + Fastify), `validation` (Python + FastAPI + OpenCV), `frontend` (Next.js). No service depends on AWS at runtime. The Claude SDK reaches the user's Max-plan auth via the `claude_home` external Docker volume — a one-time `claude setup-token` from inside the container creates the token, which then lives in the volume.

**2. Multi-agent pipeline as a typed TypeScript service.** Replace `app/nlp/` (Python + Bedrock) and `app/classifier/` (Python + Bedrock Vision) with one `app/agent/` service. Six steps form the pipeline: deterministic Input Filter (prompt-injection guard), Intent Classifier (Claude SDK Haiku 4.5), Entity Extractor (Claude SDK Haiku 4.5), deterministic Lookup (Mongo queries), Response Generator (Claude SDK Haiku 4.5 + governance prompt with few-shot examples), deterministic Output Filter (scope check). Each step appends to `interaction_logs.agent_trace[]` for audit and demo. A separate `/classify` route on the same service handles document type classification via Claude SDK Vision when n8n is gone and document upload flows back through the agent.

**3. Customer portal as a JWT-gated Next.js route.** A new `/portal/[jwt]` route on the existing frontend reads a HMAC-signed token (`{user_id, exp}`), shows the user's process timeline, interaction history with intent labels, document list with thumbnails, next-steps panel sourced from a single `visa-guidance.json` file (also used by the agent's Response Generator — single source of truth), and action buttons for handoff and "open Telegram". The token is generated by the agent service when the user types "abrir portal" via the bot, sent inline as a URL.

The work is sequenced so the system stays demoable at each phase boundary: Phase 1 leaves Sprint 3 fully working but on local infra; Phase 7 is the first phase where the new agent answers Telegram messages; Phase 9 deletes the n8n container only after the agent has fully replaced its responsibilities.

**Key data flows:**

1. **Inbound Telegram message (text):** Telegram → ngrok → `agent /telegram/webhook` → Pipeline (6 steps + logger) → Telegram `sendMessage` from agent → done. Each step writes to `interaction_logs.agent_trace[]`.

2. **Inbound Telegram message (document):** Telegram webhook → agent downloads file via Telegram API → uploads to MinIO via S3 SDK → calls `validation /validate` → if pass, calls internal `agent /classify` → Claude SDK Vision returns type → updates `files` collection in Mongo → Telegram notification sent.

3. **Status change (operator):** Operator UI → `api POST /processes/:id/status` → FSM validation in repository → `status_history` appended → API's `status-notifier` service renders deterministic template → Telegram `sendMessage` from API → done. **No LLM call. No n8n.**

4. **Customer opens portal:** User types "abrir portal" in Telegram → Pipeline routes to a deterministic action (no LLM for token gen) → JWT created from `PORTAL_SECRET` with `{user_id, exp: +24h}` → URL sent via Telegram → user clicks → Next.js middleware verifies JWT → portal page server-actions fetch processes / interaction_logs / files for that `user_id` from API.

## Tech Stack

| Layer | Tech | Why |
|---|---|---|
| Agent runtime | Node 24 + TypeScript | Direct copy of zeno-agent's Claude SDK auth pattern |
| Agent HTTP framework | **Hono** | Tiny, fast, ergonomic, used by zeno-agent |
| Agent AI SDK | `@anthropic-ai/claude-agent-sdk` ^0.2.110 | The SDK that supports the Max-plan auth flow we need |
| Agent validation | `zod` | Type-safe parsing of Telegram payloads + LLM JSON outputs |
| API runtime | TypeScript + Fastify 5 + Mongoose + tsyringe | **Unchanged** (Sprint 2 stack) |
| Validation runtime | Python 3.11 + FastAPI + uvicorn + OpenCV | Replace Lambda handler with FastAPI server, keep OpenCV core |
| Frontend | Next.js 15 + React 19 + Tailwind 4 + shadcn/ui | **Unchanged stack**, refreshed design tokens + new portal route |
| DB | `mongo:7` container | Replaces Atlas |
| Object storage | `minio/minio` container | Replaces AWS S3, S3 SDK speaks both |
| JWT | `jose` (frontend) + `jsonwebtoken` (agent) — or `jose` everywhere | HMAC HS256 |
| Tunneling | `ngrok` | Telegram webhook needs public URL |
| Tests (TS) | `vitest` | Already used by `app/api/` and zeno-agent |
| Tests (Python) | `pytest` | Standard for FastAPI |
| Linting | `biome` (TS) + `ruff` (Python) | Fast, opinionated |

## Phase Ordering

The phases are designed so each one ends with a runnable, demoable system. No phase leaves the project broken.

### Phase 1 — Local infra foundation
Add `mongo:7` and `minio/minio` to a new `docker-compose.yml`. Migrate API connection from Atlas to local Mongo. Add InteractionLog model, repository, controller, routes to `app/api/`. **End state:** API talks to local Mongo, InteractionLog endpoints exist (empty).

### Phase 2 — API status notifier (replaces n8n status workflow)
Add `telegram-notifier` and `status-notifier.service` to `app/api/`. Modify `process.controller.ts` to call status-notifier directly after FSM transition. **End state:** status changes still trigger Telegram notification, but via API directly. n8n status workflow is now dead but not deleted.

### Phase 3 — Agent service skeleton (no AI yet)
Bootstrap `app/agent/` with Hono server, env config, Mongo + MinIO + Telegram clients, JWT, health route. Dockerfile copying zeno-agent pattern. Add to `docker-compose.yml`. **End state:** `agent` container starts, `GET /health` returns OK, JWT can be signed/verified.

### Phase 4 — Multi-agent pipeline (no Telegram integration yet)
Implement all six agents + orchestrator + tracer. Unit tests for the four deterministic ones. Wire up Claude SDK auth via `claude_home` volume. **End state:** can call orchestrator with a fake user message and get a structured response with full `agent_trace[]`.

### Phase 5 — Telegram webhook + outbound (still no document handling)
Implement `POST /telegram/webhook` route on agent. Telegram client (sendMessage). Pipeline runs on incoming text → reply sent. **End state:** real Telegram messages get processed by the new pipeline (handoff also working — short-circuits before pipeline if `conversation.status === 'transferred'`). n8n's Telegram trigger is now dead but not deleted.

### Phase 6 — Document upload flow on agent
Implement Telegram getFile → MinIO upload → call `validation` → call internal classify route → notify. **End state:** user can send a photo via Telegram, agent classifies it, notification arrives. Sprint 2 document flow now goes through agent service entirely.

### Phase 7 — Validation service refactor
Replace `app/validation/`'s Lambda handler with a FastAPI server. New Dockerfile. Add to `docker-compose.yml`. **End state:** validation runs as a long-lived container, called by agent over HTTP.

### Phase 8 — Customer portal (Next.js)
Build `/portal/[token]` route. JWT verify in middleware. Reuse `process-timeline.tsx`. New components: `next-steps-panel`, `interaction-history` (with intent labels), `documents-list`, `action-buttons`. Wire `visa-guidance.json` for the next-steps panel. Add `/dashboard/interactions` operator page. **End state:** customer can open the portal via JWT URL and see everything.

### Phase 9 — Frontend UI/UX refresh
Use `ui-ux-pro-max` to design new tokens; use `frontend-design` to apply across operator console. **End state:** dashboard has the new look; portal uses the same tokens.

### Phase 10 — n8n removal + cleanup
Pre-deletion grep audit. Delete `app/n8n/`, `app/nlp/`, `app/classifier/`, the corresponding terraform modules, `scripts/generate-workflow.sh`. Update Makefile. Update `.env.example`. Verify nothing breaks. **End state:** repo has zero references to n8n / NLP Lambda / Classifier Lambda.

### Phase 11 — Tests + smoke
Vitest unit tests for filters, lookup, JWT, telegram-notifier. Integration test for `/telegram/webhook`. `scripts/smoke-e2e.ts` runs full end-to-end via real Telegram bot in a smoke account. **End state:** `npm test` passes; smoke script exits 0.

### Phase 12 — Docs + handoff
README major refresh. `RELATORIO_SPRINT_4.md` (1-2 pages). `DEMO_SPRINT_4.md` recording script. `DEV_SETUP.md` first-run guide. Update `Diagramas.drawio` + Mermaid in README. Update `bot-handoff-mechanism.md` learning with the post-n8n note. New learnings as discovered during implementation. **End state:** "tá pronto, pode gravar o vídeo" hand-off message.

## Risks / Open Decisions

**Already locked in spec — restated for the implementer:**

- **Telegram bot:** reuse `@youvisa_test_assistant_bot`. Sprint 3 demo on `sprint/3` branch is preserved if needed; Sprint 3 webhook URL is replaced when the agent registers.
- **Diagrams:** `Diagramas.drawio` is source of truth, Mermaid inline in README as readable fallback.
- **LLM model:** Claude Haiku 4.5 for all agent steps unless quality is insufficient on Response Generator (then Sonnet 4.6).

**Open at plan time — implementer must decide / ask:**

- **JWT library:** `jose` (universal, works in Node + Edge runtime) is the simplest single choice. Alternatives: `jsonwebtoken` (Node-only, simpler API). Recommend `jose` for both agent + frontend.
- **Hono vs Fastify for agent:** zeno-agent uses Hono. Recommend Hono for parity. Fastify works equally well; both have zod integrations.
- **Process FSM auto-creation from Telegram:** SPRINT_3_PLANO.md proposed it but it was never wired up (per existing learning). Sprint 4 keeps the existing behavior — operator creates manually. Auto-creation is out of scope for this sprint.
- **Operator console redesign scope:** "refresh, not rewrite" — same routes, refreshed Tailwind tokens (color palette, spacing scale, typography), polished components. Not introducing a new design system.
